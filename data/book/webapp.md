## 2. 给代码买份保险

现在你的喵宇宙已经上线，朋友们都能看到可爱的猫咪了。但玩过一会儿后，你可能会发现一个尴尬的问题：怎么大家上传的照片和编辑的信息，彼此都看不到呢？

这是因为咱们之前，照片和信息都是保存在浏览器缓存中的，这些缓存仍然在本机，互相无法共享，在编程的世界，一切信息都是数据，所以为了能互相看到彼此的信息，我们还要实现数据的共享。

要解决这个问题，我们需要做一场"大手术"：把数据从浏览器搬到云端数据库。这会涉及大量代码改动，万一改坏了怎么办？

![](/book/webapp/0c65384b07d2.png)

版本管理（Version Control）就是为了解决这个痛点而生的。在编程中，我们使用一个叫 Git 的工具来做这件事。Git 是一个免费的版本管理工具，几乎所有程序员都在用它。这是一个命令行工具，不过Claude Code 内置了 Git 支持，所以你完全不需要记忆那些复杂的命令。

### 2.1 为什么现在才学Git？

Git 的操作有点复杂，所以我们一开始没有引入，但现在是学习它的时候了——因为你正面临一个真实的风险场景。Git 就像游戏的存档功能：打 Boss 之前先存个档，失败了就读档重来。

现在我们要对代码进行"伤筋动骨"的改造，为了防止改坏了回不去，让我们启用这台"时光机"。

### 2.2 Git 的基本概念

Git 虽然有很多复杂的功能，但它的核心概念却很简单。每次你完成一个小功能，觉得"嗯，这个版本不错，值得保存"，就可以做一次"提交"（Commit），就像按下游戏的存档按钮。提交时，你可以写一段说明，比如"添加了照片上传功能"，这样以后回看历史记录时，就能知道每个版本做了什么。所有的提交会形成一条历史记录链，你可以随时查看，甚至回退到任何一个之前的版本。

![](/book/webapp/57ee3a849d24.png)

具体怎么操作呢？

假设你现在的代码已经是一个稳定的版本，我们先做个"存档"。对 CC 说：

帮我提交当前代码到 Git，提交信息写："V1.0 - 基础版本上线"

CC 会自动帮你完成所有操作——初始化 Git 仓库、添加文件、创建提交。整个过程你甚至不需要知道背后发生了什么。

如果你想查看之前做过哪些提交，可以说：

显示 Git 提交历史

你会看到一个列表，每一项都是一次提交，包含时间和说明信息：

```bash
IN: git log --oneline --decorate --graph
OUT: * 9d49a2e (HEAD -> main) V1.0 - 基础版本上线
```

万一真的需要回退到之前的版本，也很简单，告诉 CC：

回退到上一个提交

就可以了。

以后做新功能，建议养成“小步快跑”的习惯：每完成一个小改进就提交一次，甚至改个小 bug 也要做，不要等到做了很多功能才一次性提交。提交信息要写清楚做了什么，比如“添加了照片上传功能”就比“更新代码”要好得多。这样，当你需要回看历史或者回退版本时，就能快速找到想要的那个点。

Git 的能力远不止于此，它有一套很漂亮的多人协作概念，可以帮我们管理超大型的项目，想深入了解 git 使用的同学可以查看 Scott Chacon 和 Ben Strab 写的书《》或者知乎的这一篇。

### 2.2.1 与 github 的关系

看到这里，你可能会有点晕：你这两章又是git，又是github，又是 github page 的，他们到底什么关系啊？这里就解释一下：

**Git** 就像你手里的草稿本。写论文时你不会直接在正式稿纸上写，而是先在草稿本上反复修改、涂涂画画，记录每一次的改动。Git 就是代码的草稿本，它记得你删了哪段、加了哪句，改错了还能翻回之前的版本。

**GitHub** 像是老师的档案柜。你把草稿本上的定稿誊抄好，交上去存档。这样一来，哪怕你的草稿本丢了，档案柜里还有备份。而且如果有小组作业，大家都能从档案柜取阅、各自修改后再交回来。

**GitHub Pages** 则是班级门口的展示墙。老师从你交上去的论文中，挑出一部分直接贴到墙上，路过的人都能看到。你不需要再单独打印海报——只要论文在档案柜里，展示墙就会自动更新。

![](/book/webapp/c5c9bc4345e3.png)

咱们第1章先把代码上传到了github，又通过 github pages 搭建了网站，让所有人都可以访问；这一章在本地装了git，终于每一次修改都可以保存历史记录了。

## 3. 给数据找个家

现在来解决数据共享的问题。我们需要一个"云端大脑"——所有用户的猫咪信息都保存在那里，任何人访问网站时，都能从这个大脑里读取数据。

### 3.1 前后端分离架构

这时候就要引入**前后端**的概念了。简单来说，我们要把自己的程序拆成两部分，一部分用来展示，还是运行在浏览器中，称为**前端**；一部分用来存储，放在一台共享的机器上（一般叫服务器），称为**后端**。这样你和朋友们就可以通过这台服务器共享所有的猫猫照片了。

![](/book/webapp/6b50043f885c.png)

从图里可以看出：只要数据都存在中间这个服务器上，你和朋友们就可以共享所有猫猫照片了。

小贴士：你应该经常从新闻里看到，互联网企业招聘什么“前端程序员”、“后端程序员”之类的，说的就是他们各自擅长的编写在不同地方运行的程序。不过今天咱们这个 ai 打工人，可是个前后端通才，可以把所有工作都交给它。

### 3.2 数据存储

开始下一步之前，我们先思考一下，到底有哪些数据需要保存呢？

猫咪的名字

猫的信息，包括品种、年龄、性别、描述和标签

照片

这都是我们要保存的内容，哪怕不懂编程，凭直觉也能感觉到它们有些不同：名字和信息都是文本和数字信息，而照片是像素组成的图片。通常来说，保存文本和数字信息用到的技术叫数据库，而保存图片这类文件的技术叫做对象存储。

### 3.2.1 数据库存储（Database）

数据库里面保存数据的基本单元叫做表（table），它和我们平时使用的 excel 表很像，也是分为行列的，比如咱们的猫猫信息，就可以用这样的表来存储：

cats

| 编号 | 名字 | 品种 | 年龄 | 性别 | 描述 | 标签 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 小橘 | 橘猫 | 2 | 男 | 性格温顺，喜欢晒... | 温顺、亲人、吃货 |
| 2 | 雪球 | 英短 | 1 | 女 | 优雅的小公主... | 优雅、安静 |
|  |  |  |  |  |  |  |

### 3.2.2 文件存储（File Storage）

文件存储，也叫对象存储，它非常简单，就是前端可以把文件上传，服务器接收这个文件以后，保存起来，并且给它分配一个 url  地址，这样所有人就都能从浏览器中访问这个文件了。URL 大概长这个样子：`https://vswrhnmhumfxgfrwpfwc.supabase.co/storage/v1/object/public/cat-images/6/1762661050243_tw65sc.png`

你可能会有一点疑惑：这个 url，跟我们在 Github Page 上部署自己的网页，获取的 url 很像啊，只是长一点？对，实际上他们的存储和访问技术也是很像的，HTML 页面也是一个文件嘛。不过它们的使用场景还是有些不同：

之前我们的 HTML 页面，是由程序员（就是你呀）编写并上传的，一般用户（比如你的朋友们）只是访问和查看。通常上传不会很频繁，毕竟需要编码更新了才会有一次上传嘛；总的文件数也不会太多；所以通过 git action 来处理就足够了，哪怕普通用命令行上传也完全够用

而对象存储的内容不同：设想一下，如果咱们的“喵宇宙”项目有了千百万的用户，每时每刻都有人想上传自己爱喵的照片，就需要有专门的技术来应对这种非常频繁又大量的文件保存了，这就是对象存储的意义。

仔细看上面的 url 的话，还会发现：上传后的文件名，都有很长的一串字符`1762661050243_tw65sc.png`，看上去没什么意义，这也是对象存储的标准做法：通过特定的算法，保证上传后的文件，有自己“唯一”的名字，这样成千上万的文件之间就不会重名。

小贴士：我们以后看到技术相关的文章，可能会遇到一个词儿，叫数据持久化，其实它就是我们前面提到的数据存储——因为只有存起来，数据才不会消失，下次启动程序的时候，还能看到它们。这个词儿，是从作用的角度来表达数据存储的。从这个意义上说：之前我们提到的浏览器本地存储，本章节讲到的数据库和文件，都属于数据持久化的范畴，只是持久化的位置不同。

### 3.3 选择 Supabase

经典编程模式中，我们需要写后端程序来实现对数据库和对象存储的控制，然后再开放 API 接口给前端程序，相互沟通，整个过程变得比较复杂（当然也是有价值的，我们最终会把项目变成这种模式，但目前还不需要）。

对于很多练习性的小项目来说，我们难免会想：就不能让前端程序直接连接数据存储么？可以的，有需求就有供给：一些云服务厂商，就提供了这样面向互联网，前端可以直接访问的存储服务，其中的佼佼者就是 supabase。可以说，几乎所有 vibe coding 的云服务商，都接入了 supabase 的数据存储能力，它就是 ai 编程时代的标配。

### 3.3.1 创建 Supabase 项目

于是我们得到类似这样一段语句：

```sql
-- 创建猫咪信息表
CREATE TABLE cats (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, -- 编号
    name varchar NOT NULL,   -- 名字
    breed varchar, -- 品种
    sex smallint CHECK (sex IN (1, 2)), -- 1=公, 2=母
    age numeric,  -- 年龄
    tags varchar, -- 逗号分隔的标签字符串
    description text, -- 描述
    images text[], -- 图片 URL 列表
    created_at timestamp DEFAULT NOW(),  -- 创建时间
    updated_at timestamp DEFAULT NOW()   -- 更新时间
);
```

从英文的含义也可以看出，这里给每个列起了个名字，并且告诉数据库，它们的类型，下面的给出了详细的解释，现在不想看也没问题，可以略过：

id：编号，是每条数据的唯一标识（也就是提示词里的“主键”），这里我们选择用简单的数字来表示。

name：猫猫的名字，类型选择 varchar，就是长度不确定的文字，后面字数不定，又不是太长的文字，都可以用这个类型。这里多了一个 NOT NULL 标识，表示在插入一行数据的时候，名字必须得有，不允许记录一只没有名字的猫猫。

breed：品种，也选择 varchar。（要想支持用户选择品种的话，这里还需要更复杂的设计，不过目前暂时保存名字就好了）

age：年龄，类型选择 numeric，因为年龄不一定是整数，这里就不选 int 开头的类型了。

sex：性别，类型选择 int2。你可能有点奇怪，男/女不是文本么，为什么要选数字类型？这其实是常见的习惯，当可选项只有两个的时候，用 0 和 1 来表示，会简单清晰（什么？你觉得性别有 128 种？这不在本项目的考虑范围）。

description：描述，类型选择 text，这里不用 varchar，是因为描述有可能很长，而且通常不会用描述的文字来查找

tags：标签，类型选择 varchar

created_at：创建时间，一般我们都会想记录一下数据是什么时候添加到数据库的，它有个缺省值 now()，说明我们在保存数据的时候，不需要自己设置这个字段的值，supabase 数据库会自动用当时的时间来存储

updated_at：和创建时间类似，只不过记录的是数据变更的时间

images：用来保存照片的位置（注意不是照片文件，文件本身是要放在对象存储的），它的类型有点特别，虽然选择了 text，齿轮图标激活的窗口里，还要选中”Define as Array“，表示可以存储多张照片的位置：

### 4.2.2 关于安全

细心的你，如果看了上面访问数据库的代码，可能会想："把密钥直接写在代码里，这安全吗？"
这是个好问题。Supabase 提供的是 **Anon Key**（匿名密钥），它本身就是为了公开使用设计的，只能访问你允许公开的数据。但即便如此，养成"环境变量"的意识也很重要。
在真实的生产项目中，我们会把这类配置放在环境变量里，而不是直接写在代码中。虽然前端代码最终还是会暴露这些密钥（因为浏览器需要它们来发起请求），但这种做法至少让配置管理更清晰，也方便在不同环境（开发/测试/生产）之间切换。

目前我们使用公开访问策略（RLS 未启用）来快速验证功能。在实际产品中，你应该配置行级安全策略（RLS），限制用户只能访问自己的数据。我们稍后就会做这个改进。

### 4.2.4.2 第二步：追踪一个完整流程

知道了整体结构，接下来我们跟着一个具体的功能走一遍，看看代码是怎么协作的。我们选择最基础的场景：用户打开页面，看到猫咪卡片。

这时候，你可以这样问 AI：

我想理解用户从打开页面到看到猫咪卡片的完整流程，能一步步给我讲解吗？每一步都告诉我：发生了什么、为什么要这样做、如果这里出错了会有什么现象。

**AI**：好的，让我们跟随一位叫小美的用户，看看她打开你的应用后会发生什么：

> **第 1 步：浏览器加载页面**

小美的浏览器开始下载你的 `index.html` 文件。这时候就像剧院开门，观众入场。如果这一步出问题（比如文件找不到），她会看到"404 页面不存在"的错误。

> **第 2 步：解析 HTML 和 CSS**

浏览器读取 HTML，搭建页面骨架；读取 CSS，给页面穿上"衣服"。这时候页面看起来已经成型了，但还没有数据。如果 CSS 加载失败，页面会变成"素颜"——所有元素挤在一起，没有颜色。

> **第 3 步：检查登录状态**

JavaScript 开始运行，首先问数据库："这位访客之前来过吗？"如果小美登录过，右上角会显示"欢迎回来"；如果没登录，显示"登录/注册"按钮。这一步出问题，可能会看到"加载中"一直转圈。

> **第 4 步：获取猫咪数据**

JavaScript 向数据库发送请求："请给我所有猫咪的信息。"数据库返回一个列表，包含每只猫的名字、年龄、照片等。如果这里出问题（比如网络断了、数据库连接失败），页面会显示"暂无猫咪"或空白。

> **第 5 步：渲染卡片**

JavaScript 拿到数据后，开始"组装"卡片。它为每只猫创建一个卡片元素，填入照片、名字、描述，然后插入到页面上。这时候小美终于看到猫咪了！如果这里出错，可能看到数据但排版混乱。

> **第 6 步：绑定交互事件**

最后，JavaScript 给每个按钮"注入灵魂"——点赞按钮知道被点击后要做什么、评论框知道怎么弹出。这一步出问题，页面看起来正常，但点击按钮没反应。

### 4.2.6 图片上传：保存到对象存储

用同样的方法，共享图片的存储。首先编辑 prd.md，增加下面两句：

```markdown

- Public bucket（公开访问）
- 文件组织：`{猫咪id}/{当前时间}_{随机字符串}.{文件扩展名}`
```

首先确认存储桶的名字，是我们之前创建的`cat-images`，然后指定文件名。因为对象存储本质上，就是在上传/下载文件，而上传后的文件，不会保留原来的名字了（要不然所有人都上传，存储文件夹就很容易重名），所以这里我们给上传后的文件名，定了个规矩：都是按照`猫咪 id`做文件夹，`当前时间 + 随机字符串`做文件名。

然后再跟 CC 说：

现在，可以把照片存储，也修改为存在 supabase storage 了

于是文件上传的代码也改变了，变成了下面这个函数：

```javascript
        /**
         * Upload image to Supabase Storage
         * 上传图片到 Supabase Storage
         * @param {File} file - Image file
         * @param {number} catId - Cat ID
         * @returns {Promise<string>} - Public URL of uploaded image
         */
        async function uploadImageToStorage(file, catId) {
            if (!supabaseEnabled || !supabase) {
                console.error('uploadImageToStorage: Supabase 未启用');
                throw new Error('Supabase 未启用');
            }

            // Generate unique filename - 生成唯一文件名
            const timestamp = Date.now();
            const fileExt = file.name.split('.').pop();
            const randomStr = Math.random().toString(36).substring(7);
            const fileName = `${timestamp}_${randomStr}.${fileExt}`;
            const filePath = `${catId}/${fileName}`;

            console.log(`uploadImageToStorage: 文件路径 = ${filePath}`);
            console.log(`uploadImageToStorage: 文件大小 = ${file.size} bytes`);
            console.log(`uploadImageToStorage: 文件类型 = ${file.type}`);

            // Upload file - 上传文件
            console.log('uploadImageToStorage: 调用 supabase.storage.upload()...');
            const { data, error } = await supabase.storage
                .from('cat-images')
                .upload(filePath, file, {
                    cacheControl: '3600',
                    upsert: false
                });

            if (error) {
                console.error('uploadImageToStorage: 上传失败', error);
                console.error('uploadImageToStorage: 错误详情:', JSON.stringify(error, null, 2));
                throw error;
            }

            console.log('uploadImageToStorage: 上传成功', data);

            // Get public URL - 获取公开 URL
            const { data: urlData } = supabase.storage
                .from('cat-images')
                .getPublicUrl(filePath);

            const publicUrl = urlData.publicUrl;
            console.log(`uploadImageToStorage: 公开 URL = ${publicUrl}`);

            return publicUrl;
        }
```

核心就是这一句：

```bash
const { data, error } = await supabase.storage
                .from('cat-images')
                .upload(filePath, file, {
                    cacheControl: '3600',
                    upsert: false
                });
```

### 4.3 项目演进的思路

写到这里，我想和你聊聊 vibe coding 里的一个核心心法——怎么让项目稳稳地往前推进，而不是在某个环节把自己绕晕。

你可能已经感觉到了：我们每一次都是发现项目的一个小小的不足，然后把它补齐，而不会一次性实现所有的功能。

这就是 vibe coding 最重要的规律：**一次只攻一个山头**。

比如刚才连接数据库这件事，第一轮对话我们先让代码跑通——能从数据库读出数据、能显示在页面上，就算胜利。这时候页面可能很简陋，错误处理也不完善，甚至 Loading 状态都没有，没关系，第一轮的目标就是"通"。

等到基本流程跑顺了，第二轮我们再回来打磨细节：加个旋转的 Loading 动画，处理一下网络失败的情况，优化一下图片加载的体验。每一轮都是在上一次的基础上做 refinement（细化），而不是另起炉灶搞新功能。

还有一种情况特别常见——代码改着改着，突然发现之前的思路有问题，或者 AI 生成了太多代码，你已经看不太懂了。这时候别硬撑，**关掉当前对话，新开一个 session**。把刚才验证过的、确定好用的版本作为起点，把要做的改进说清楚，重新开始。这就像下棋时推倒重摆，不是为了否定之前的努力，而是让大脑清空缓存，轻装上阵。

这种"小步快跑、频繁存档"的节奏，和传统的软件开发很不一样。以前我们习惯先做完整的需求分析、画架构图、写技术方案，一切都想清楚了才动手。但 vibe coding 的乐趣恰恰在于**边做边想**——让 AI 快速把想法变成看得见的东西，然后根据实际效果决定下一步往哪走。

说到这你可能会问：怎么判断一个功能是不是"做完了"？我的标准是：**能用就行**。Loading 状态有了，错误提示有了，基本的交互逻辑通了，就可以先放一放。 perfectionism（完美主义）是 vibe coding 的大敌，因为明天你很可能又有新想法，或者发现更好的实现方式。

现在我们的喵宇宙已经能读写数据库了，这是一个重要的里程碑。先停下来，把玩一下，确认没什么大问题，然后再往下走——去搞定用户登录和权限管理。这种稳扎稳打的感觉，比一口气冲到底然后被一堆 bug 淹没，要舒服多了。

### 6.2 RLS：产品级安全

RLS（行级安全）是 PostgreSQL 数据库的一个强大功能。简单来说，它让数据库在返回数据前先问一句："这个用户有权看这条记录吗？"

我们的策略配置是：

所有人都能查看所有猫咪（社交属性）

但只有猫咪的主人才能修改或删除

这是一种从"完全开放"到"精细控制"的安全升级。在实际产品中，你可以根据需求配置更复杂的策略，比如：

允许管理员管理所有数据

允许用户设置某些内容为私密

限制用户每天只能发布一定数量的内容

## 7. 颜值即正义

到现在，喵宇宙的功能已经相当完整了。但你可能已经审美疲劳了——好吧，现在的页面真的称不上美，不但配色单一，看起来有一股浓浓的"AI 味儿"，而且也没有针对手机显示优化过。

### 7.1 响应式设计：适配移动端

既然要把链接发给朋友看，就得考虑他们多半是用**手机**打开的。你可能记得我们在入门篇创建第一个页面的时候，它就是可以在手机端展示的，不过后面呢又做了那么多修改，我们从来也没有针对手机端再做过优化，如果你跟着教程走到现在，很可能它在手机上已经面目全非了。当然如果你运气好的话，没准看起来还不错——这是强大的 ai 模型在背后默默替我们操心。

无论如何，咱们心里要有个数：页面不是自动就能在手机端展示的很好的，它需要一种响应式设计来适配。

响应式设计（Responsive Design）就是让页面在不同设备上都能良好显示的技术。
最基础的响应式设计使用 **Media Queries**，说白了就是不同大小屏幕展示成不同的样式，类似下面这样：

```css
/* 默认样式：手机端 */
.cards-grid {
    display: grid;
    grid-template-columns: 1fr; /* 单列 */
    gap: 16px;
}

/* 平板端：屏幕宽度大于 768px */
@media (min-width: 768px) {
    .cards-grid {
        grid-template-columns: repeat(2, 1fr); /* 两列 */
    }
}

/* 桌面端：屏幕宽度大于 1024px */
@media (min-width: 1024px) {
    .cards-grid {
        grid-template-columns: repeat(3, 1fr); /* 三列 */
    }
}
```

### 7.2.3 修复现场：样式改了，功能怎么崩了？

UI 重构后，我发现页面确实变好看了，但点击"编辑"按钮却没有任何反应。"是不是重构的时候把 JavaScript 搞坏了？"

打开浏览器的开发者工具（F12），切换到 Console 面板，发现一条黄色警告：

```javascript
Uncaught TypeError: Cannot read property 'classList' of null
    at HTMLButtonElement.editCat
```

这时候不要慌，也不要急着回滚所有改动。把问题抛给 CC：

点击编辑按钮没有反应，控制台显示：

```

Uncaught TypeError: Cannot read property 'classList' of null
    at HTMLButtonElement.editCat

```

帮我检查一下是什么原因。

CC 会快速定位到问题所在：原来是元素被嵌套进新的容器后原来的相对定位逻辑失效了。修复通常很简单——要么恢复原来的 DOM 结构，要么更新 JS 中的选择器路径。
