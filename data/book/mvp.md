在前面的《准备篇》中，你已经安装好了环境，包括 Claude Code（简称 cc）和 VSCode 对应的插件。

3. **限定条件充足**：

功能：倒计时到天、时、分、秒

视觉：深蓝渐变背景、白色大字体

特效：烟花动画

适配：手机和电脑

细节：数字大而清晰

更换不同的背景色，比如表示喜庆的大红色

### 1.2 错误处理三板斧

做练习的时候，你可能会发现，ai 也会犯错——让它实现一个背景音乐，结果连原来的烟花都显示不出来了，面对空无一物的页面，你有点欲哭无泪。别着急，我们需要跟页面打配合，告诉它发生了什么。最简单的方法，就是描述一下遇到的现象：

现在什么都显示不出来了，页面一片空白

cc 收到后，会重新阅读一遍代码，从中找出可能的问题（专业的说法，叫做代码 review）。不过这显然不是那么容易——因为我们没告诉它任何具体的错误信息。cc 盲目地从所有代码中猜测可能的问题，又慢又低效。

这时候，就要祭出我们的 vibe coding 调试三板斧了：

1.2.1 **第一招：看控制台**

对于我们文字描述的错误，cc 需要猜测原因，很多情况下，不如直接告诉它错误信息，更容易定位。查看这个错误信息的地方，就叫叫做控制台（console），它内置在我们的浏览器中。不同的浏览器，会有不同的打开方式。比如我们这里使用的 Chrome，就点击右上角三个点的按钮，在展开的菜单中选择：更多工具 -> 开发者工具。（或者按 F12）

查看“开发者工具”也是程序员调试的必备技能。

**打开开发者工具**

浏览器窗口，按 F12，或者右键点击页面，选择"检查"(Inspect)，或者在菜单栏下面的位置，都可以找到它。

**找到 Console 标签**

打开后会看到一排标签，找到标记为 "Console" 的选项（通常在 "Elements"、"Sources" 等标签旁边），点击它。

![](/book/mvp/5af317877cea.png)

如果程序出错了，这里通常会显示一些红色字体的信息，这些红色错误就是浏览器在报告说"我这里有 xxx问题"。

把它们复制下来，发给 cc

我的页面出错了，控制台显示：
"""
Uncaught NotAllowedError: play() failed because the user didn't interact with the document first.
"""
这是为什么呢？

**注意**，在复制多行引用内容的时候，可以用一组符号，把他们跟你自己的话隔离开，就像例子中的 `"""`（或者用反引号 ``` 也可以），这是一种让阅读更清晰的纯文字"排版"方式——把 ai 当做一个人，适合人阅读的，也适合 ai。

这段错误信息，对 ai 定位问题非常有帮助，它甚至会因此而感谢你：

你提供的信息太重要了！这个错误是因为我在页面加载时添加了自动播放音乐的代码，但现代浏览器出于用户体验考虑，不允许网页在没有用户交互的情况下自动播放带声音的媒体！

然后给出解决方案：

修改改为：页面上放一个“播放音乐”按钮，或者让用户点击任意区域后再播放，可以么？

在征得你的同意之后，ai 会自动修复它。

1.2.2 **第二招：贴图**

并不是所有的错误，都能在控制台找到错误信息的，很多时候，只是因为我们的指令不太清晰，被 ai 误解，做出来的效果不符合预期。

比如我在做这个练习的时候：

把倒计时目标，改到端午节

Cc 快速完成了修改，甚至还添加了非常漂亮的鲤鱼图标，但是左上角却出现一个莫名其妙的问号：

![](/book/mvp/578223601ebd.png)

这时如果只能口头描述页面的实际情况，和需求的差异，是不太容易说清楚的，好在现在的 ai 模型，大部分都自带了“眼睛”，可以看到我们提供的图像。

我们就可以截图 + 标注错误点之后，粘贴给 cc：

![](/book/mvp/ec0c47689b7b.png)

Cc 回答：

问题已经解决了！问号是因为之前的emoji编码有问题。我已经修复了：

**标题**：改为"2026 端午节倒计时"

**页面标题**：使用🥟（粽子）emoji替换了有问题的🎏 emoji

**目标日期**：设置为2026年6月17日（端午节）

现在倒计时页面应该正常显示，从2026年2月1日到6月17日还有136天的倒计时，没有问号问题了。页面会显示天、小时、分钟和秒的实时倒计时，还有漂亮的烟花动画效果。

1.2.3 **第三招：持续迭代**

刚才的修改明显还有问题，我们从对话中也能看出来图标不太对，那么还可以继续让它修改：

你这是粽子么？这是饺子啊

对不起，我理解错了，但是没有粽子的 emoji 图标，我用改竹子🎋代表了，这样既有端午节的特色，又能准确表达。

![](/book/mvp/dcc2f0b4763c.png)

类似的问题，可能会经常出现，我们需要一些耐心，一次次反馈错误信息，持续调教，直到满足需求为止。你会发现，虽然目前的 ai 已经很强了，cc 又是最好用的 vibe coding 工具，它还是经常犯低级的错误。这是因为在本质上，ai 还没有长期记忆的能力，所有的信息，其实都保存在一次会话的聊天记录当中，聊得长了，就难免捡起这个，忘了那个，甚至前后内容夹杂不清。

好在我们这个初体验项目比较简单，怎么样才能更好地管理聊天记录，会在后面的实战项目中讲到。

2. **实战项目：喵宇宙**

在上一节里，你体验了用一句话生成小项目的乐趣，也顺便学习了一点错误反馈的技巧。但这些项目都是“一次性”的——生成后就完成了。现在，我们要做一个**真正的项目：**从最简单的版本开始，一步步添加功能，最终部署上线，成为一个可以分享给朋友的应用。

2.1 **做什么**

**喵宇宙**是一个在线猫咪展示平台。你可以为猫咪创建专属卡片，上传照片，创建个性化展示空间。做好的展示卡片可以轻松分享到微信、微博和朋友圈，在电脑和手机上随时查看。

2.1.1 **为什么选这个项目**

**有趣**

猫咪是人类最好的朋友之一，人人都爱晒猫，哪怕你自己没有养，也可以获得很多朋友的喜爱。你还可以和朋友比谁在社区中发现其他人的猫咪，收获满满的快乐。

这个项目的学习曲线非常友好，只需5分钟你就能创建第一个猫咪卡片，10分钟上传自己的猫咪照片，30分钟做出精美的展示页面，1小时后就能拥有完整的展示馆。后续还可以添加特效让猫咪更萌，可以将项目部署上线，让全世界都能访问。

**实用**

这个项目做完之后，真的可以用起来，展示自己家的喵星人，把它可以分享给朋友(建立猫友社区)，还可以举一反三创作更多的作品(狗狗展示馆、宝宝成长册、旅行相册等)。

### 2.1. 需求定义

跟之前的一句话项目不同，咱们这个喵宇宙稍稍繁杂一些，还准备上线以后可以长期提供服务。所以在正式开始编码之前，你需要做一些准备工作。就像盖房子要先画图纸、准备工具一样，做项目也需要先定义需求、准备环境。

**2.1.1 什么是 PRD(产品需求文档)?**

PRD 是 Product Requirements Document 的缩写，用来描述"我们要做什么"。

你可能会想：**“我不是在和 AI 对话吗？为什么还要写文档？”**

当项目变得复杂后，一句话说不清楚所有需求，文档能帮你理清思路，避免遗漏重要功能。更重要的是，文档是给 AI 看的“产品说明书”，无论开多少个会话，AI 都能看着文档生成符合你需要的代码。

**好消息**：PRD 不需要很复杂，用自然语言写清楚就行！

**2.1.2 MVP 版本的 PRD**

MVP 是 Minimum Viable Product（最小可行产品）的缩写，指的是**最简单但能用的版本**。

采用 MVP 的方式开发有很多好处：它能让你快速看到成果，获得成就感和反馈；避免一开始就陷入复杂功能的泥潭；更重要的是，你可以先验证核心想法是否可行，再决定是否投入更多时间。就像搭积木，我们先搭好基础框架，确保它稳固，然后再一层层添加更多精彩的部分。

对于喵宇宙项目，我们先从最简单的版本开始：一个展示猫咪信息的卡片。

**PRD 文档示例(阶段 1 MVP 版本)**：

```text
代码块
### 项目名称
* 喵宇宙 —— 猫咪卡片展示

### 功能描述
* 一个展示猫咪的信息和照片的网页。

### 核心功能
* 显示多张精美的猫咪卡片
* 卡片包含：猫咪名字、年龄、性格描述
* 卡片设计精美，有阴影和圆角效果

### 技术要求
* 所有代码放在一个 HTML 文件中，名字叫做 index.html
* 纯前端实现，无需后端
```

2.2.3 **原理解释**

你可能好奇背后的原理。简单来说，浏览器提供了一种叫 File API 的能力，让网页可以读取你电脑上的文件。当你选择一张照片后，浏览器会创建一个临时的图片链接，然后代码把这个链接设置给卡片上的图片元素。下面是 AI 可能生成的代码片段，看看就好，现在不用完全理解（后面我们会删除这个调用 /手动狗头）

```javascript
代码块
// 文件选择按钮的事件监听
function processAndDisplayImage(file, imgElement, catId) {
    const reader = new FileReader();

    // Read file and convert to data URL - 读取文件并转换为数据URL
    reader.onload = (e) => {
        const imageUrl = e.target.result;

        // Update image source - 更新图片源
        // CSS object-fit: cover will handle cropping automatically
        // CSS的object-fit: cover会自动处理裁剪
        imgElement.src = imageUrl;

        // Store in localStorage for persistence - 存储到本地存储以持久化
        try {
            localStorage.setItem(`cat-image-${catId}`, imageUrl);
        } catch (e) {
            console.warn('无法保存图片到本地存储:', e);
        }
    };

    reader.onerror = () => {
        alert('文件读取失败，请重试');
    };

    // Read file as data URL - 读取文件为数据URL
    reader.readAsDataURL(file);
}

/**
 * Load saved images from localStorage
 * 从本地存储加载保存的图片
 */
function loadSavedImages() {
    catsData.forEach(cat => {
        const savedImage = localStorage.getItem(`cat-image-${cat.id}`);
        if (savedImage) {
            const imgElement = document.querySelector(`.cat-image[data-cat-id="${cat.id}"]`);
            if (imgElement) {
                imgElement.src = savedImage;
            }
        }
    });
}
```

2.3.3 **阶段 5：分享动图**

添加一个按钮，

## 总结

恭喜你！跟随本章一路走来，已经完成了一个令人惊叹的转变：**从零编程经验，到独立创建自己的 Web 页面。**

回顾这段旅程。你从一个简单的新年倒计时页面开始，通过几轮对话就让烟花在深蓝色的渐变背景上绽放，数字精准地跳动着倒计时。然后你完成了更复杂的喵宇宙项目，从最初只显示几张猫咪卡片，到支持上传自己的照片，再到实现多图轮播、信息编辑，最后还能导出精美的动图分享给朋友。这两个项目看起来简单，但背后涉及的技术细节其实相当丰富——只不过 AI 帮你处理了那些复杂的部分。

更重要的是，你掌握了 Vibe Coding 的核心方法。你学会了用“言出法随”的方式，用自然语言向 AI 描述需求，让它为你编写代码。你体会到了迭代的力量，不再追求一次就做到完美，而是通过多轮对话逐步完善作品。当遇到问题时，你知道该打开浏览器控制台查看错误信息，知道可以截图标注问题点，也懂得了要给 AI 足够的耐心和反馈，让它一步步调整到符合你的期望。

在这个过程中，你还学会了如何清晰地表达需求。动词要明确（“做一个”而不是“想要”)，名词要具体（“新年倒计时网页”而不是“倒计时”），限定条件要充足（要说明功能、视觉、特效、适配等各方面的要求）。你也掌握了在同一会话中持续改进的技巧，理解了 AI 的局限性——它很博学却容易"健忘"，需要我们不断地提供上下文和反馈。

**Vibe Coding 的本质**

通过这一章的实践，你应该已经感受到：**Vibe Coding 并不是让 AI 完全替你编程，而是人机协作的创造过程。**你提供创意、需求和反馈，AI 负责实现细节和处理技术问题，然后你们一起迭代打磨，最终创造出独一无二的应用。

这个过程中，你不需要记住各种编程语法，不需要背诵复杂的 API 文档，但你需要知道自己想要什么。清晰的目标是一切的起点。你还需要能用自然语言准确地描述需求，这种沟通能力比编程技能更重要。同时你要有敏锐的判断力，能发现哪里不对劲，能给出具体的反馈。最后，你需要足够的耐心，愿意一步步迭代完善，而不是指望一次就完美。

**当前的限制**

不过坦白说，现在你手中的作品还有一些明显的局限。它只能在你自己的电脑上运行，朋友想看的话，你得把整个 HTML 文件发给他们。所有数据都保存在浏览器的本地存储里，不仅容量极其有限(只有 5MB)，而且一旦清理浏览器缓存就会全部丢失。你的应用也没有用户系统，每个人看到的都是同样的内容，无法实现个性化。更重要的是，这些应用缺少真正的"后台"，所有逻辑都在前端，无法处理复杂的业务场景。

这些限制，正是我们在进阶章即将突破的目标。
