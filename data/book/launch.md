## 海外与国内部署区别

想象你刚做完一份独门披萨，迫不及待想让世界尝尝。这时候你来到一条“**全球美食街**”：不需要任何营业执照、平台免费给你起步用的炉子、桌子，你只需带上食材，一键开店（部署），顾客从世界各地都能吃到你的披萨（网站全球加速访问）。虽然美食街在海外，离中国顾客可能有点远，送餐（访问）稍慢，但至少披萨已经能被全世界吃到啦。

等你名气渐长、顾客越来越多，就该考虑在国内商场开实体店，这里的条件会严格些:你得先办**营业执照**（ICP 备案）、得配合**消防检查**（实名域名、内容审核），得到的结果是国内的访问更稳定、配送更快。国内云厂商（阿里云、腾讯云、华为云）就是这种“商场房东”。他们给你一个稳定摊位、合规流程，但需要你多花点时间装修。国内部署的服务可以更好的可以触达14亿的广袤市场，国内部分我们会在进阶篇论述。

**CI/CD**

**CI（Continuous Integration，持续集成）** 和 **CD（Continuous Deployment，持续部署）** 听起来很高大上，其实用披萨店的话说，就是一套“自动化质检与出货系统”。

### 角色分配

**CI（持续集成）—— 自动质检员**： 每当你修改了代码并 `git push` 到 GitHub 时，CI 就会自动跳出来。它会帮你检查代码有没有写错（语法检查）、逻辑通不通（运行测试）。

**比喻**：每当你切好一盘新火腿，质检员会自动尝一块。如果变质了（代码报错），它会立刻拉响警报，不准这盘火腿进入烤箱。

**CD（持续部署）—— 自动送餐员**： 只要 CI 质检通过，CD 就会接过接力棒，自动把代码打包、构建、发布到服务器上，替换旧版本。

**比喻**：披萨烤好了，自动打包机直接把它封箱并交给无人机，瞬间送到顾客桌上。你全程不需要动一根手指。

### 为什么你需要它？

**拒绝“手动搬运”**：你不再需要用 FTP 或者手动拖传文件。那种“改个错别字要折腾半小时”的石器时代已经过去了。

**降低“翻车”概率**：CI 会在你上线前帮你发现 bug。如果代码跑不通，部署会自动停止，老版本的网页会继续正常运行，不会让用户看到一个挂掉的页面。

**小步快跑**：你可以一天上线 10 次！哪怕只是改了一个按钮的颜色，改完 `push` 一下，3 分钟后全世界看到的都是新版。

DNS 记录就像互联网的 “地址簿”，把好记的域名翻译成电脑能识别的 IP 地址。

**核心作用**

不用记一串难背的 IP（比如 192.168.1.1），输入就能找到对应服务器。

负责域名和 IP 的 “配对管理”，还能实现分流、备份等功能。

**常见的几种 DNS 记录**

A 记录：最常用的 “直接配对”，把域名指向 IPv4 地址（比如把指向 123.45.67.89）。

AAAA 记录：和 A 记录类似，只是指向 IPv6 地址（新一代互联网地址）。

CNAME 记录：“别名记录”，让一个域名跳转到另一个域名（比如让指向）。

MX 记录：“邮件专用记录”，告诉邮件服务器该把邮件送到哪个地址（比如 xxx@xxx.com 的接收服务器）。

TXT 记录：“备注记录”，可添加验证信息、防垃圾邮件说明等（比如证明你是域名的所有者）。

## 在云上正式部署你的应用

刚刚我们聊完了Docker，大家对部署上线应该有了一些了解，接下来就是选择自己的云平台部署应用了，按照 **“抽象程度”**（即：你作为开发者需要管多少事）从高到低进行分类。抽象程度越高，越自动化和AI化；抽象程度越低，自主控制权越大，您需要根据项目情况、个人使用的技术等情况，妥善进行平台选择。

云生态百花齐放、类型各异，较为常见的平台的不完全分类如下：

**AI 驱动/全自动生成型** **：**你不需要写代码，只需要写 prompt（提示词）。平台负责生成代码、预览、部署。一般而言，**你无法在此类平台部署你本地的代码，只能用自然语言进行软件生成**。适用原型验证、着陆页、简单工具、非技术人员开发。此类玩家主要有v0 (by Vercel)/Lovable / Bolt.new等，这是目前最火的赛道。它们可以进行全栈代码生成--不仅生成 UI，还能在浏览器里运行全栈应用，一键部署。

它们的优点是全自动，缺点是缺乏项目掌控，例如v0生成的项目，如果你连接了supabse这个平台存储数据，supabase将进入托管模式，很多操作会无法运行。

![](/book/launch/3ef0c4d9c57f.png)

**前端托管 / 边缘网络型 **：专注于 Web 前端。你的代码使用Git Push后即部署，还会自动配置 SSL、CDN、域名。它们**也能跑后端**（Serverless Functions），但核心基因是前端。适用个人博客、公司官网、Next.js/Vue/React 等网页应用。其中玩家主要有：前端托管的王者Vercel，Next.js 的亲爹。体验极致丝滑。Vercel 的老对手Netlify，功能非常全面，免费额度很大方。

如果要部署前端页面，大概率您会和他们打交道，又或者是使用其他的容器等平台部署前端，缺点就是只能使用它们规定的技术做后端，而且长时间运行的任务价格昂贵。

**后端即服务型**：平台帮你搞定了**数据库、用户认证 (Auth)、文件存储、实时订阅**。你只需要写前端，直接调用它们的 SDK 就能存取数据。你不需要“部署”后端，因为后端已经在那了。适用全栈应用、小程序、不想写 CRUD 后端的开发者。其中的玩家是：

**Supabase:** 开源版 Firebase 的最强替代者。基于 PostgreSQL，提供超强的数据库能力 + Auth + Edge Functions。

**Firebase:** Google 旗下，老牌霸主，实时数据库依然很强，但绑定 Google 生态较深。

**Appwrite:** 另一个优秀的开源 BaaS，功能类似 Supabase，Docker 部署很方便，云上免费版给了很多额度，缺点是功能不那么完善。

上述后端服务平台会存在一定的**技术绑定**问题：在方便的同时，如果您用了平台开发，后续就很难迁移到别的平台或者技术去了。当然，如果您仅使用其一部分能力，例如您只用Supabase的数据库能力，那么迁移还是简单的（如supabase，它的数据库是标准的pg库）。

![](/book/launch/1529697d76a8.png)

**容器托管平台 / 现代 PaaS** ：这就是你刚才学的 Docker 发挥作用的地方。你给平台一个 Docker 镜像（或者让平台检测你的代码自动构建），它帮你跑起来。它比 VPS 简单，比 Vercel 灵活（可以跑 Python、Go、Java、长时间运行的任务）。适用传统的后端服务、Docker 应用、微服务、数据库托管，主要玩家有：

**Railway:** 极简主义，UI 极其漂亮。自动识别代码语言，支持 Dockerfile。

**Render:** 类似于 Railway，提供免费的 Web Service（有休眠机制）和数据库，非常适合练手。

**Zeabur (泽布):** 国人开发的优秀平台。

上述平台的好处就是，用了它们和您电脑上安装、使用Docker一样的体验。

**在线集成开发环境**：它连代码编辑器都在云端。买的是“开发环境 + 运行托管环境”。适用教学、快速验证想法、不想配置本地环境、Chromebook 用户。比较出名的是Replit**，**它给你一个 Linux 容器，你在浏览器里写代码、按 Run 就能跑，现在它正通过 AI 向全自动进化。其也存在少量的技术绑定问题：文件存储、kv数据库等一些服务只能在云端replit运行，如果和其生态绑定过深，你的代码很可能没法从replit上迁移。

**基础设施即服务** (IaaS / VPS) ： 给你一台裸 Linux 虚拟机（甚至是一台真的服务器！）。你需要做 SSH 登录，安装 Docker，配置 Nginx，配置防火墙，甚至自己装数据库。自由度 100%，但难度 100%。适用复杂的大型应用、需要极致省钱、需要特殊系统配置、学习 Linux。主要玩家有AWS EC2 / Azure / Google Cloud也就是大家常说的“大厂云”。Vultr**:** 对开发者友好的云厂商、阿里云 / 腾讯云**:** 国内备案必选。

总结：自动化程度越高的平台和技术，其核心好处在于**大幅降低开发门槛与启动成本** —— 无需手动搭建服务器、配置环境或从零开发后端 API，甚至**非专业开发者也能借助 AI 功能参与开发，让初创团队或个人可快速验证原型并聚焦核心业务**；同时，平台整合 “编码 - 测试 - 部署” 全流程，能将传统数天的上线周期压缩至分钟级，适配互联网产品快速迭代需求，且承担了底层运维工作。

但此类平台的不足集中在**局限性与隐性成本**上：主要是供应商锁定风险，闭源平台的专有 API 与数据格式会增加迁移难度，即便开源平台（ Supabase），深度依赖定制功能也会提升适配成本；免费版资源限制明显，无论是 Replit 的 CPU 配额、Firebase 的函数调用次数，还是 v0.app 的生成次数，**都难以支撑复杂项目**，**而企业级功能的付费套餐长期使用成本可能超过自建基础设施**。

因此从全自动的v0到全手动的云服务器，大家在选择的时候需要认真考虑

**先跑起来再说**：新手别纠结"最优解"，先用最简单的方式部署上线，跑通流程最重要。

**从高抽象往低抽象走**：先用 Vercel / Railway 这类平台，等你真正遇到它们的限制了，再考虑迁移到 VPS。

**注意锁定风险**：如果用了某个平台的专有功能，想清楚未来迁移的成本。

**混合使用要谨慎**：前端用 Vercel + 后端用 Railway + 数据库用 Supabase，听起来很美好，但每个平台都要学习、每个平台都有坑、每个平台都可能收费。

## 最后的叮嘱：部署成本问题

部署上线是开心事，但**账单可能让你不开心**。一定要算好部署成本！！！这里有几个血泪教训，提前打个预防针：

1、免费额度往往有限：各大平台的免费额度看起来很香，但细看条款会发现，当你的应用稍微复杂起来之后，免费额度就超了，**不管用什么平台都一定要看好是怎么收费的**。

2、惊喜账单：这是最坑的一个。你的项目突然火了，流量暴涨，例如有开发者的 Vercel 项目被爬虫刷了，一夜之间产生了 **$10,000+** 的账单。所以需要设置账单告警，超过阈值立刻通知你。最好设置用量上限，到了上限自动停服，别让它无限烧钱。

3、混合使用不同平台要谨慎，会导致成本叠加问题：这是我个人的血泪教训。为了"各取所长"，我曾经用过这样的组合：

**v0** 生成前端 → 付费

**Vercel** 托管前端 → Pro 版付费

**Supabase** 做数据库和 Auth → Pro 版付费

**Railway** 跑后端长任务 → 付费

听起来很专业对吧？**一个月账单加起来将近 $100**，换成人民就很贵了。
